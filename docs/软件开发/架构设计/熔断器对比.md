# 熔断器对比：Sentinel vs Hystrix vs Resilience4j

> 来源：个人笔记整理 | 标签：架构 / 高可用 / 熔断

---

## 三大熔断框架

| 框架 | 开发者 | 状态 | 特点 |
|------|--------|------|------|
| **Sentinel** | Alibaba | 活跃 | 面向云原生，功能最全 |
| **Hystrix** | Netflix | 维护中（已停止新功能） | 线程池隔离 |
| **Resilience4j** | 独立 | 活跃 | 轻量级，Spring推荐 |

---

## 功能对比

| 特性 | Sentinel | Hystrix | Resilience4j |
|------|----------|---------|--------------|
| **隔离策略** | 信号量隔离 | 线程池隔离/信号量隔离 | 信号量隔离 |
| **熔断降级** | 基于响应时间、异常比率、异常数 | 基于异常比率 | 基于异常比率、响应时间 |
| **实时统计** | 滑动窗口（LeapArray） | 滑动窗口（RxJava） | Ring Bit Buffer |
| **动态规则** | 多种数据源 | 多种数据源 | 有限支持 |
| **限流** | 基于QPS，支持调用关系限流 | 有限支持 | Rate Limiter |
| **流量整形** | 预热、匀速器、排队 | 不支持 | 简单限流 |
| **系统自适应** | ✅ 支持 | ❌ 不支持 | ❌ 不支持 |
| **控制台** | 开箱即用，实时监控 | 简单监控 | 无控制台 |
| **扩展性** | SPI扩展点 | 插件形式 | 接口形式 |

---

## Sentinel 核心优势

- 承接阿里巴巴近10年双十一流量防护场景
- 丰富的流量控制场景（秒杀、削峰填谷、集群流量控制）
- 完善的控制台（规则配置、秒级监控、机器发现）
- 对 Dubbo、Spring Cloud、Web Servlet、gRPC 等无缝集成

## Resilience4j 适用场景

- 轻量级，适合Java 8+函数式编程
- 无外部依赖（仅使用Vavr）
- 不适合复杂企业级服务架构
- 缺乏生产级控制台

## 熔断模式对比

### Sentinel
```java
@SentinelResource(value = "resource")
public Result getData() {
    // 流量控制 + 熔断降级
}
```

### Hystrix（线程池隔离）
```java
@HystrixCommand(threadPoolKey = "pool1",
    threadPoolProperties = @HystrixProperty(name = "coreSize", value = "10"))
public Result getData() {
    // 线程池隔离
}
```

### Resilience4j（信号量隔离）
```java
@CircuitBreaker(name = "backendA", fallbackMethod = "fallback")
public String getData() {
    return backendService.doSomething();
}
```

---

## Resilience4j 核心组件

| 组件 | 作用 |
|------|------|
| **Retry** | 重试失败的请求 |
| **CircuitBreaker** | 熔断器，快速失败 |
| **RateLimiter** | 限流 |
| **TimeLimiter** | 超时限制 |
| **Bulkhead** | 隔离（限制并发） |
| **Cache** | 缓存成功结果 |
| **Fallback** | 失败后备方案 |

---

## 选型建议

- **云原生微服务**：选 Sentinel（功能最全+控制台）
- **Spring Cloud Netflix 体系**：Hystrix（已被Spring Cloud移除，可选Resilience4j）
- **轻量级/新项目**：Resilience4j（无外部依赖）
