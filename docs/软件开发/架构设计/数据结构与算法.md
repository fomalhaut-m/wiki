## 为什么需要数据结构

数据结构的主要目的是**更方便地存储和更高性能地查找**。

| 数据结构 | 查询复杂度 | 特点 |
|----------|-----------|------|
| 数组/二分查找 | O(log n) | 需要排序 |
| 链表 | O(n) | 插入快，查询慢 |
| 跳表 | O(log n) | 链表+随机抽取中心点优化 |
| 平衡二叉树 | O(log n) | 树高导致磁盘IO多 |
| **B树** | O(log n) | 多路平衡，减少IO |
| **B+树** | O(log n) | MySQL默认索引结构 |

## 二叉树遍历

每个节点在搜索路径上会出现**三次**，访问时机决定了遍历方式：

### 三种遍历方式

| 遍历 | 顺序 | 示例 |
|------|------|------|
| **先序遍历** | 根 → 左 → 右 | ABDECF |
| **中序遍历** | 左 → 根 → 右 | DBEACF |
| **后序遍历** | 左 → 右 → 根 | DEBFCA |

## B树 vs B+树

### B树特点
- 多路平衡查找树
- 每个节点可以有多个子节点
- 所有节点都存储数据

### B+树特点
- **叶子节点存储所有数据**
- 叶子节点之间有**链表指针**（便于范围查询）
- 非叶子节点只存储键

### MySQL 选 B+树的原因

| 原因 | 说明 |
|------|------|
| **方便扫库** | B树需要中序遍历，B+树直接扫叶子链表 |
| **支持区间查询** | B+树叶节点链表串起，便于 range-query |
| **减少IO** | 非叶子节点不存数据，树高更矮 |

### B+树分裂
当节点满时，分配新节点，原节点1/2数据复制到新节点，父节点增加新指针。

## 算法评价

### 时间复杂度（从小到大）

| 复杂度 | 名称 |
|--------|------|
| O(1) | 常数阶 |
| O(log n) | 对数阶 |
| O(n) | 线性阶 |
| O(n log n) | 线性对数阶 |
| O(n²) | 平方阶 |
| O(2ⁿ) | 指数阶 |

### 空间复杂度

算法所需的存储空间，包括：
- 程序本身所用空间
- 输入数据占用空间
- 辅助变量占用空间
