# TCP 协议详解

> 来源：TCP详解笔记 | 标签：网络 / TCP

---

## TCP 特点

- **面向连接**：传送数据前必须建立连接
- **可靠**：保证数据有序、不丢失
- **基于字节流**：全双工模式
- **可靠连接**：三次握手建立连接

---

## TCP 首部

### 四要素确定唯一连接

```
（源IP，源端口）+（目的IP，目的端口）
```

- 端口范围：0-65535（1024以下系统保留）

### 序号和确认号

| 字段 | 说明 |
|------|------|
| **seq** | 32位序号，当前传输方向字节流的每个字节序号 |
| **ack** | 32位确认号，对上一次seq的确认，ack=seq+1 |

---

## TCP 标志位

| 标志 | 说明 |
|------|------|
| **SYN** | 同步标志位，建立连接，同步序列号 |
| **ACK** | 确认标志位，对已接收数据确认 |
| **FIN** | 完成标志位，关闭连接 |
| **PSH** | 推送标志位，接收后立即交上层应用 |
| **RST** | 重置标志位，连接复位、拒绝非法包 |
| **URG** | 紧急标志位，紧急指针域有效 |

---

## 三次握手

```
客户端                  服务器
  │                       │
  │──── SYN=1, seq=J ────→│  第一次握手（客户端发送SYN）
  │                       │
  │←── SYN=1, ACK=1, ────│  第二次握手（服务器发送SYN+ACK）
  │      seq=K, ack=J+1   │
  │                       │
  │──── ACK=1, ack=K+1 ─→│  第三次握手（客户端发送ACK）
  │                       │
  │     ESTABLISHED       │  连接建立成功
```

### 状态变化

| 状态 | 说明 |
|------|------|
| `SYN_SENT` | 第一次握手后，客户端等待 |
| `SYN_RCVD` | 第二次握手后，服务器等待 |
| `ESTABLISHED` | 三次握手完成，连接建立 |

## 四次挥手

```
客户端                  服务器
  │                       │
  │──── FIN=1, seq=M ────→│  第一次挥手（客户端请求关闭）
  │                       │
  │←── ACK=1, ack=M+1 ───│  第二次挥手（服务器确认）
  │                       │
  │   （客户端等待服务器   │
  │    关闭连接）         │
  │                       │
  │←── FIN=1, seq=N ─────│  第三次挥手（服务器请求关闭）
  │                       │
  │──── ACK=1, ack=N+1 ─→│  第四次挥手（客户端确认）
  │                       │
  │     TIME_WAIT         │  客户端等待（2MSL）
  │                       │
  │     连接关闭           │
```

### 为什么需要四次挥手？

TCP是全双工模式，关闭连接需要双向分别关闭：
1. 客户端发送FIN，请求关闭客户端→服务器方向
2. 服务器ACK确认
3. 服务器发送FIN，请求关闭服务器→客户端方向
4. 客户端ACK确认

## 常见问题

### 为什么需要三次握手？

防止失效的连接请求导致服务器资源浪费。如果只用两次握手，失效的请求会被服务器当作新连接建立，导致空等。

### 为什么挥手比握手多一次？

握手时：SYN+SYN/ACK合并为一次  
挥手时：FIN不能与ACK合并（FIN代表关闭方向）
