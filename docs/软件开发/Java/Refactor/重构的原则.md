# 重构的原则

> 来源：重构读书笔记 | 标签：代码质量 / 重构

---

## 何为重构

**重构（名词）**：对软件内部结构的调整，**不改变外观行为**，提高可理解性，降低修改成本。

**重构（动词）**：使用重构手法，**在不改变可观察行为**的前提下调整结构。

> 如果重构期间代码有一两天不可用，那他做的不是重构。

---

## 两顶帽子

开发时时间分配两种行为：

| 行为 | 做什么 | 不做什么 |
|------|--------|----------|
| **添加新功能** | 只管添加新功能 | 不修改既有代码 |
| **重构** | 调整代码结构 | 不添加新功能 |

---

## 为何重构

1. **改进软件设计** — 防止架构腐败变质
2. **使软件更容易理解** — 代码可读性
3. **帮助找Bug** — 深入理解代码，发现隐藏问题
4. **提高编程速度** — 良好结构容易修改，bug少

---

## 何时重构

> 事不过三，三则重构

| 时机 | 说明 |
|------|------|
| **预备性重构** | 添加新功能前，先调整代码使工作更容易 |
| **帮助理解的重构** | 把自己的理解转移到代码（变量名、长函数拆分） |
| **捡垃圾式重构** | 发现做的不好的代码，但不想在当前任务跑题 |

---

## 重构常见手法

| 手法 | 说明 |
|------|------|
| 提取函数 | 把长函数拆成短函数 |
| 变量改名 | 使变量名更具描述性 |
| 提取变量 | 把复杂表达式赋值给清晰变量 |
| 搬移函数 | 把函数放到更合适的类/模块 |
| 合并重复代码 | 消除重复 |

---

## 坏味道清单

- 重复代码
- 长函数
- 过大类
- 过长参数列表
- 发散式变化（一个类被多种原因修改）
- 散弹式修改（一种变化需修改多个类）
- 依恋情节（类过分依赖其他类）
- 数据泥团（多处出现相同数据组合）
- 基本类型偏执（不愿用小对象）
- 魔法数字
- 过短类/过短函数
