# 重构知识体系

> 来源：《重构》Martin Fowler | 标签：编程实践 / 重构

---

## 重构两顶帽子

| 状态 | 做什么 |
|------|--------|
| **添加新功能** | 不改既有代码，只管添加 |
| **重构** | 不添加新功能，只调整结构 |

> 如果重构期间代码不可用，那就不叫重构。

---

## 为何重构

1. **改进设计** — 防止架构腐败变质
2. **易于理解** — 代码自解释
3. **帮助找Bug** — 深入理解代码，发现问题
4. **提高速度** — 良好内部结构让修改更容易

---

## 何时重构

> **事不过三，三则重构**

1. **预备性重构** — 添加功能前先微调
2. **理解性重构** — 让代码更易读（改名、拆分函数）
3. **捡垃圾式重构** — 发现问题立即改或记录
4. **长期重构** — 逐步解决大问题和团队达成共识

## 代码的坏味道

### 必须重构的信号

| 坏味道 | 说明 | 解决方案 |
|--------|------|----------|
| **神秘命名** | 名称不表明意图 | 改名 |
| **重复代码** | 多处相同代码 | 提炼函数 |
| **过长函数** | 函数太长难理解 | 拆分（找注释、条件、循环处） |
| **过长参数列表** | 参数过多 | 保持对象完整、合并参数 |
| **全局数据** | 任何地方都可修改 | 封装变量 |
| **可变数据** | 变量在别处被修改 | 封装变量、查询/修改分离 |
| **发散式变化** | 一个修改要改多个地方 | 拆分阶段、搬移函数 |
| **散弹式修改** | 一种修改要改多个类 | 组合函数成类 |
| **依恋情节** | 函数对另一个模块交流更频繁 | 搬移函数 |
| **数据泥团** | 相同数据出现在多处 | 提炼类、引入参数对象 |
| **基本类型偏执** | 不用对象封装基本类型 | 以对象取代基本类型 |
| **重复switch** | 多处相同switch | 多态取代条件表达式 |
| **循环语句** | 鄙视循环 | 管道取代循环 |
| **冗赘元素** | 只有函数没有类 | 内联函数、内联类 |
| **夸夸其谈通用性** | 用不到就不需要 | 简化 |
| **临时字段** | 字段仅为特定情况设 | 提炼类 |

## 重构手法（常用）

| 重构 | 说明 |
|------|------|
| 提炼函数 | 长函数拆成小函数 |
| 内联函数 | 短函数合并 |
| 搬移函数 | 函数移到正确的类 |
| 提炼类 | 职责过多的类拆分 |
| 引入参数对象 | 参数列表过长时合并 |
| 以对象取代基本类型 | 基本类型封装为类 |
| 拆分阶段 | 不同阶段分离 |
| 封装变量 | 限制直接访问 |
