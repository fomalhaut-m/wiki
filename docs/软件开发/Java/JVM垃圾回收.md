# JVM 垃圾回收器

> 来源：垃圾回收笔记 | 标签：JVM / GC

---

## 堆内存分区（JDK 1.8前）

```
堆内存 = 新生代 + 老年代 + 永久代
新生代 = Eden + Survivor1 + Survivor2
```

> JDK 1.8 移除了永久代，取而代之的是**元空间（Metaspace）**，直接使用物理内存。

---

## 为什么要垃圾回收

- 内存不是无限的，分配后不回收迟早会耗尽
- Java 语言没有显式释放内存的方法（`System.gc()` 只是**建议**JVM，不是立即执行）
- GC 自动监控对象引用情况，回收不再使用的对象

---

## 触发 GC 的时机

1. **年轻代/老年代满了** — 无法为新对象分配内存
2. **手动调用 `System.gc()`** — 通常触发一次 Full GC + 至少一次 Minor GC
3. **守护线程轮询** — 低优先级 GC 线程自动运行

---

## Minor GC vs Full GC

| 类型 | 发生区域 | 频率 | 速度 |
|------|----------|------|------|
| **Minor GC** | 新生代 | 频繁 | 快 |
| **Full GC** | 老年代 | 较慢 | 慢（慢10倍以上） |

---

## 并行 vs 并发

| 模式 | 用户线程 | GC线程 | 代表 |
|------|----------|--------|------|
| **并行（Parallel）** | 等待 | 多条并行工作 | ParNew、Parallel Scavenge |
| **并发（Concurrent）** | 继续运行 | 与用户线程同时执行 | CMS、G1 |

---

## 七种垃圾收集器

### 新生代收集器

| 收集器 | 线程 | 算法 | 特点 |
|--------|------|------|------|
| **Serial** | 单线程 | 复制 | 简单高效，Client模式默认 |
| **ParNew** | 多线程并行 | 复制 | Serial的多线程版本 |
| **Parallel Scavenge** | 多线程并行 | 复制 | 目标是吞吐量优先 |

### 老年代收集器

| 收集器 | 线程 | 算法 | 特点 |
|--------|------|------|------|
| **Serial Old** | 单线程 | 标记-整理 | Server模式备选 |
| **Parallel Old** | 多线程并行 | 标记-整理 | 吞吐量优先 |
| **CMS** | 并发 | 标记-清除 | 最短停顿时间，4步流程 |

### JDK 11+ 新收集器

| 收集器 | 特点 |
|--------|------|
| **ZGC** | 低延迟，可处理TB级堆，停顿时间<10ms |

---

## CMS 四步流程

```
1. 初始标记（CMS initial mark）— 标记GC Roots直接关联的对象（停顿）
2. 并发标记（CMS concurrent mark）— GCRoots tracing（不停顿）
3. 重新标记（CMS remark）— 修正并发标记期间的对象变动（停顿）
4. 并发清除（CMS concurrent sweep）— 清理垃圾（不停顿）
```

> 优点：并发收集、低停顿
> 缺点：产生浮动垃圾、空间碎片

---

## 吞吐量

```
吞吐量 = 运行用户代码时间 / (运行用户代码时间 + GC时间)
```

例如：100分钟运行，GC花1分钟 → 吞吐量 = 99%

---

## G1（Garbage First）

- 将堆划分为多个大小相等的 Region
- 优先回收价值最大的 Region（垃圾最多的区域）
- 目标：吞吐量与停顿时间的平衡
- JDK 9+ 成为默认垃圾收集器
