# 重构：改善既有代码的设计

> 来源：《重构》读书笔记 | 标签：代码质量 / 重构

---

## 何为重构

**重构（名词）**：对软件内部结构调整，**不改变可观察行为**，提高可理解性，降低修改成本。

**重构（动词）**：使用一系列重构手法，**在不改变行为**的前提下调整结构。

> 如果重构期间代码一两天不可用，那他做的不是重构。

---

## 两顶帽子

| 行为 | 做什么 | 不做什么 |
|------|--------|----------|
| **添加新功能** | 只管添加 | 不修改既有代码 |
| **重构** | 调整结构 | 不添加新功能 |

---

## 为何重构

1. 改进软件设计 — 防止架构腐败
2. 使软件更容易理解
3. 帮助找 Bug — 深入理解代码行为
4. 提高编程速度 — 良好结构容易修改

---

## 何时重构

> 事不过三，三则重构

| 时机 | 说明 |
|------|------|
| **预备性重构** | 添加新功能前先微调代码 |
| **帮助理解的重构** | 把理解转移到代码（改名、拆函数） |
| **捡垃圾式重构** | 发现不好的代码，趁还在上下文时处理 |
| **长期重构** | 分阶段逐步解决大问题 |
| **复审时重构** | 代码复审 + 重构结合 |

---

## 重构挑战

| 挑战 | 对策 |
|------|------|
| 延缓新功能 | 重构是为了更快开发，必要时才做 |
| 代码所有权 | 开源型团队，鼓励跨团队修改 |
| 分支合并难 | 持续集成（CI），每天向主线集成 |
| 遗留代码 | 每次碰触时改善一点点 |
| 数据库 | 渐进式数据库设计，分多次迁移 |

---

## 重构与性能

1. 良好代码 = 有充裕时间进行性能调整
2. 清晰代码 = 更细颗粒度性能分析
3. 好理解 = 更好选择调整方案

---

## YAGNI

**YAGNI** = You Aren't Going to Need It

简单设计、增量式设计，待问题理解充分后再着手解决。
