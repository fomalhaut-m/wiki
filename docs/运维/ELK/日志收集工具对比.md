# 日志收集工具对比

## 三款主流工具

| 工具 | 语言 | 特点 | 适合场景 |
|------|------|------|---------|
| **Logstash** | JRuby（JVM） | 插件最丰富，社区活跃，功能强大 | 复杂日志解析，大规模架构 |
| **Fluentd** | CRuby + C | 轻量，设计简洁，性能好 | 容器环境（K8s 默认集成） |
| **Filebeat** | Go | 最轻量，部署简单 | 只做日志采集，配合 Logstash 使用 |

> **阿里云 Logtail**：C++ 实现，专注稳定性和资源控制，适合阿里云用户。

## 功能对比

| 功能 | Logstash | Fluentd | Filebeat | Logtail |
|------|----------|---------|----------|---------|
| 日志读取方式 | 轮询 | 轮询 | 轮询 | **事件触发** |
| 文件轮转 | 支持 | 支持 | 支持 | 支持 |
| Failover | 支持 | 支持 | 支持 | 支持 |
| Grok 正则解析 | 强 | 弱 | 不支持 | 弱 |
| 数据过滤 | 支持 | 支持 | 不支持 | 支持 |
| 数据缓冲 | 插件支持 | 插件支持 | 不支持 | 支持 |
| 热升级 | 不支持 | 不支持 | 不支持 | **支持** |
| 中心化配置 | 不支持 | 不支持 | 不支持 | **支持** |
| 运行环境 | JVM | Ruby | Go | C++ |
| 多线程 | 支持 | 受 GIL 限制 | 支持 | 支持 |
| CPU/内存保护 | 无 | 无 | 无 | **支持阈值保护** |

## 选型建议

```
轻量采集（推荐 Filebeat）
    ↓
需要复杂解析 → Logstash
    ↓
K8s 容器环境 → Fluentd
    ↓
阿里云用户 → Logtail
```

## Filebeat vs Logstash 分工

生产环境中，两者通常配合使用：

```
Filebeat（采集）→ Logstash（解析过滤）→ Elasticsearch（存储）→ Kibana（展示）
```

- **Filebeat 负责**：采集日志、初步处理、发送到下游
- **Logstash 负责**：深度解析、正则过滤、字段转换

这样可以减轻 Logstash 的资源压力，因为大部分日志文件读取由 Filebeat 完成。
