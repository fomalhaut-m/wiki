# 消息队列

> 来源：个人笔记整理 | 标签：架构 / 消息队列

---

## 为什么需要MQ

1. **异步处理** — 减少等待时间
2. **削峰填谷** — 缓解高并发压力
3. **解耦** — 生产者和消费者松耦合
4. **可靠传输** — 消息持久化

---

## RabbitMQ

### 核心概念

| 概念 | 说明 |
|------|------|
| Producer | 生产者 |
| Consumer | 消费者 |
| Exchange | 交换机（路由消息） |
| Queue | 队列（存储消息） |
| Binding | 绑定（Exchange和Queue的关联） |

### 消息模式

**点对点**：一条消息只能被一个消费者消费

**发布订阅**：一条消息被所有订阅者消费

### Exchange类型

| 类型 | 说明 |
|------|------|
| **direct** | 完全匹配路由键 |
| **fanout** | 广播给所有队列 |
| **topic** | 按通配符匹配 |
| **headers** | 按消息头匹配 |

---

## Kafka

### 特点

- 高吞吐量（百万级TPS）
- 分布式持久化
- 支持消息回溯
- 实时流处理

### 核心概念

| 概念 | 说明 |
|------|------|
| Topic | 消息主题 |
| Partition | 分区（并行消费） |
| Producer | 生产者 |
| Consumer | 消费者 |
| Consumer Group | 消费者组 |

### 适用场景

- 日志收集
- 实时流处理
- 事件驱动架构
- 大数据分析

---

## 对比

| 对比项 | RabbitMQ | Kafka |
|--------|----------|-------|
| 吞吐量 | 中等 | 极高 |
| 延迟 | 低 | 毫秒级 |
| 消息持久化 | 支持 | 支持 |
| 消息回溯 | 不支持 | 支持 |
| 适用场景 | 企业级异步消息 | 日志、流处理 |

---

## 使用注意

1. **消息丢失** — 确认机制（Ack）
2. **消息重复** — 幂等处理
3. **顺序性** — 队列内部有序，分区保证并行
4. **死信队列** — 异常消息处理
