# AKF Scale Cube 扩展模型

> 来源：AKF Partners | 标签：架构 / 扩展性

---

## 概念

Scale Cube（AKF Scale Cube）是分割服务、定义微服务和扩展产品的模型。

三个扩展维度：
- **X 轴**：服务和数据的水平复制和克隆
- **Y 轴**：功能分解和分段（微服务化）
- **Z 轴**：沿客户边界的服务和数据分区（分片）

---

## X 轴扩展

**方式**：负载均衡器后面运行应用程序的多个相同副本。

| 优点 | 缺点 |
|------|------|
| 实现简单 | 必须完整复制数据集 |
| 提高应用程序容量和可用性 | 会话状态难以分发 |
| 事务扩展性好 | 运营成本高 |

---

## Y 轴扩展

**方式**：沿名词或动词边界分离服务和数据（功能分解）。

常见拆分：
- 从浏览中拆分搜索
- 从添加到购物车的结账
- 订单管理 vs 客户管理 vs 库存

| 优点 | 缺点 |
|------|------|
| 故障隔离（每个服务独立数据源） | 需要团队有服务所有权 |
| 缓存命中率提高 | 运维复杂度增加 |
| 运营成本降低 | |
| 组织可扩展性提高 | |

---

## Z 轴扩展

**方式**：沿客户边界分割（按 customer_id、地理等）。

常见拆分：
- 按 customer_id 无偏模数
- 按地理边界（美国 pod、欧盟 pod，符合 GDPR）

| 优点 | 缺点 |
|------|------|
| 故障隔离 | 软件相同但数据不同，组织可扩展性不增加 |
| 缓存命中率提高 | |
| 运营成本降低 | |
| 符合当地隐私法规 | |

---

## 三轴对比

| 轴 | 拆分维度 | 目的 |
|---|---------|------|
| **X** | 水平复制 | 提高容量和可用性 |
| **Y** | 功能分解 | 提高组织可扩展性、故障隔离 |
| **Z** | 客户/地理分区 | 符合法规、提高响应时间 |

**实际架构通常同时使用三个轴。**
